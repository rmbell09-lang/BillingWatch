BillingWatch v1.0: Self-Hosted Stripe Anomaly Detection

📅 April 3, 2026 · ⏱ 5 min read

launch stripe open-source self-hosted

BillingWatch v1.0 is out. It's a self-hosted Stripe webhook monitor that catches billing anomalies before they become customer complaints, chargebacks, or silent revenue leaks.

If your SaaS runs on Stripe, weird things happen constantly: duplicate charges, payment failures that don't retry, subscriptions that quietly lapse, events that arrive out of order. Most of the time you find out from a customer or a panicked Slack message — not from your own monitoring. BillingWatch fixes that.

The problem it solves

Stripe sends webhooks for everything: payments, subscriptions, refunds, disputes. The problem is that Stripe itself doesn't tell you when something looks wrong — it just fires the event. Catching anomalies requires you to build the logic to compare events over time, track state, detect patterns.

Most teams don't build that. They check Stripe's dashboard manually, write one-off scripts, or get alerted by customers. BillingWatch builds that logic once and runs it for you, locally, on your own infrastructure.

7 detectors, zero ongoing cost

BillingWatch ships with seven anomaly detectors out of the box:

Duplicate charge detector
Flags charges for the same customer within a configurable time window
Payment failure spike
Alerts when failure rate exceeds your baseline within a rolling period
Subscription lapse monitor
Catches subscriptions that cancel without a corresponding customer action
Out-of-order event detector
Detects webhook delivery ordering issues that can corrupt subscription state
Revenue drop alert
Compares MRR snapshots across periods and flags drops beyond threshold
Refund anomaly detector
Flags unusual refund volumes, partial refund patterns, or refund spikes
Invoice discrepancy check
Catches invoices where line items don't match the subscribed plan price

Each detector is configurable per tenant. The entire system runs on SQLite + FastAPI — no external database, no cloud dependency, no per-event pricing.

Multi-tenant from day one

BillingWatch is built for agencies and developers who run Stripe for multiple clients. Each tenant gets isolated event storage, separate detector thresholds, and independent alert configuration. One deployment handles all of them.

Architecture: FastAPI backend · SQLite storage · Webhook signature verification · REST API for tenant management · JSON digest reports · Docker + Render deploy-ready

Getting started

Clone the repo and start receiving webhooks in under 5 minutes:

git clone https://github.com/rmbell09-lang/BillingWatch
cd BillingWatch
cp .env.example .env  # add STRIPE_WEBHOOK_SECRET
docker compose up

Then point a Stripe webhook endpoint at https://your-host/webhooks/stripe and BillingWatch starts monitoring. Use the REST API to configure tenants, query anomalies, and set alert thresholds.

Full setup docs are in the repo: github.com/rmbell09-lang/BillingWatch

What's next

The v1.0 focus was correctness and reliability — getting the detectors right and keeping the deployment simple. On the roadmap:

  • Email and Slack digest delivery (webhook → alert → you)
  • Dashboard UI for anomaly browsing and false positive tuning
  • Playwright integration test suite against Stripe's test mode
  • Additional detectors: trial conversion gaps, proration errors

If you're running Stripe and want a billing safety net that you control, BillingWatch is worth 5 minutes to set up.
